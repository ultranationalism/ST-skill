# 完全前端卡构建约定与未验证的注意事项

本文档记录设计/构建完全前端卡时**实际采用的约定**，以及一些**观察到但未充分复现的可疑行为**。和 API 文档不同，这里的内容没有经过系统性验证——读者应当把它们当作「以防万一的工程默认」而不是「ST 确定存在的特性」。

---

## 关于 JS 源码中的反引号（模板字面量）

### 当前约定

仓库内示例卡 `cards/inspection-station/` 的**所有** JS 源文件**都不使用模板字面量**：

- 10 个 `src/*.js` 文件里反引号数量 = 0
- 所有字符串拼接统一用 `'...' + var + '...'` 或 `[...].join('\n')`
- `build.cjs` 在构建期额外做一个**弱检查**：统计 fragment 内反引号总数，若为奇数则报警（`build.cjs:169-175`）

```javascript
// 推荐写法
const msg = '你好 ' + name + '，欢迎来到 ' + place;
const html = '<div class="' + cls + '">' + content + '</div>';

const prompt = [
  '你是一个叙事生成器。',
  '当前场景：' + scene,
  '玩家动作：' + action,
].join('\n');
```

### 为什么采用这个约定

**动机是谨慎而非已证实的 bug**。最终的卡被 ` ```html ` fence 包裹后会流经 ST 的 markdown → 渲染管线，过程中至少有以下环节**可能**干扰反引号：

- markdown 围栏匹配：行首三连反引号会提前关闭代码块
- ST 自身的宏/regex/模板替换机制
- 未配对的反引号破坏 fence 完整性

**这些风险并未在当前环境下被系统性复现**。仓库里 `tmp/fix-backticks.cjs` 是一个一次性脚本，曾把某张卡的所有模板字面量转成字符串拼接，但转换的**动机和真正解决的问题**在 commit 历史里没有留下明确记录。它有可能解决的是其中某一个具体问题，也有可能是「以防万一」的预防性清理。

### 如何看待这个约定

- **把它当成工程默认**：字符串拼接比模板字面量多几个字符，但避免了一整类潜在的渲染/解析问题，收益/成本比合理
- **不要把它当成 ST 的已证特性**：如果未来有人需要用模板字面量（比如引入第三方库，其打包产物必然包含反引号），**应当实际测试**再决定如何处理，而不是假设「反引号一定坏」
- **弱检查不是真守护**：`build.cjs` 的奇数反引号检查只能发现一种极端情况（fence 破坏）。偶数个反引号（2、4、6...）是模板字面量的常态，会完全绕过检查。如果要把工具当成保险，需要进一步加强检测（例如直接统计 `\`...\`` 模式或 `${` 出现次数）

### 复现/调研建议（如果未来需要弄清楚真相）

1. 做一张最小卡：JS 里只放一行 `console.log(\`hello ${name}\`)`，其他保持常规结构
2. 分别测试以下变体：
   - 模板字面量 + 无 `${...}` 插值
   - 模板字面量 + 有 `${...}` 插值
   - 字符串拼接（作为对照组）
3. 观察现象：HTML 是否正常渲染为 iframe？JS 是否正常执行？console 是否有报错？
4. 若不同变体表现不同，记录**在哪一环**被破坏（DOMPurify？markdown 渲染？ST 宏替换？iframe 沙盒？）
5. 把结论写回本文档，同时评估是否值得放宽约定

在此之前，保持「JS 源码不用模板字面量」作为默认工程规范。
